// 01 — Définition Fondamentale

C'est quoi exactement ces projets ?

Avant de rentrer dans la technique, comprendre le problème qu'ils résolvent.

📝

LOGGING

Enregistrer ce qui se passe dans ton application — chaque action, erreur, transaction — dans des fichiers ou bases de données structurés.

  • Qui a fait quoi, quand, avec quel résultat
  • Erreurs avec stack trace complète
  • Temps de réponse par requête
  • Événements métier (paiement, connexion…)
📡

MONITORING

Observer en temps réel l'état de santé de tes systèmes — CPU, mémoire, latence, erreurs — et déclencher des alertes automatiques.

  • Métriques temps réel (CPU, RAM, JVM…)
  • Alertes automatiques si seuil dépassé
  • Détection d'anomalies
  • Uptime / Downtime tracking
📊

REPORTING

Transformer les logs et métriques en dashboards visuels, rapports PDF/Excel, ou e-mails automatiques pour les équipes métier.

  • Dashboards interactifs Grafana
  • Rapports planifiés (quotidien, hebdo)
  • KPIs business pour le management
  • Export PDF / Excel automatisé
💡
Le problème réel résolu : Imagine ton app en prod tombe à 3h du matin. Sans logging/monitoring, tu ne sais PAS pourquoi, tu ne peux PAS reproduire le bug, tu ne peux PAS savoir combien d'utilisateurs ont été affectés. Ces projets t'apprennent à avoir une visibilité totale sur tes systèmes — c'est ce que font toutes les grandes entreprises (Netflix, Google, Amazon).

// 03 — Logging Deep Dive

Le Logging en profondeur

Tout ce que tu dois savoir sur comment Spring Boot logue et comment le structurer.

01

📦 Les Frameworks de Logging SLF4J Logback

Spring Boot utilise SLF4J comme façade (interface abstraite) et Logback comme implémentation par défaut. Pourquoi ? Parce que tu veux pouvoir changer d'implémentation (Log4j2, etc.) sans changer ton code.

// UserService.java — Logging basique import org.slf4j.Logger; import org.slf4j.LoggerFactory; @Service public class UserService { // Un Logger par classe — convention universelle private static final Logger log = LoggerFactory.getLogger(UserService.class); public User createUser(UserDto dto) { log.info("Creating user: {}", dto.getEmail()); // INFO = flux normal try { User user = userRepository.save(mapper.toEntity(dto)); log.info("User created successfully. id={}, email={}", user.getId(), user.getEmail()); return user; } catch (Exception e) { log.error("Failed to create user: {}", dto.getEmail(), e); // ERROR = problème throw e; } } }

Niveaux de log (du plus au moins important)

ERROR Exception, crash, bug critique
WARN Situation anormale mais récupérée
INFO Événement normal important
DEBUG Détails de débogage
TRACE Hyper-verbeux, chaque étape

Configuration logback-spring.xml

<!-- logback-spring.xml --> <appender name="FILE" class="RollingFileAppender"> <file>logs/app.log</file> <rollingPolicy class="TimeBasedRollingPolicy"> <!-- Nouveau fichier chaque jour --> <fileNamePattern>app.%d{yyyy-MM-dd}.log</fileNamePattern> </rollingPolicy> </appender>
02

🔑 Structured Logging (JSON Logs) Logstash Encoder

Au lieu de logs texte classiques, on logue en JSON. Pourquoi ? Parce qu'Elasticsearch peut les indexer et chercher dessus en millisecondes.

❌ Log texte classique (difficile à parser)

2024-01-15 14:32:01 ERROR UserService - Failed to create user: john@example.com java.lang.RuntimeException: DB timeout at UserRepository.save(...)

✅ Log JSON structuré (searchable)

{ "timestamp": "2024-01-15T14:32:01Z", "level": "ERROR", "service": "user-service", "traceId": "abc123def456", "userId": "john@example.com", "message": "Failed to create user", "exception": "DB timeout" }
03

🗺️ MDC — Mapped Diagnostic Context Correlation IDs

Le MDC permet d'attacher des informations contextuelles (userId, requestId, sessionId) à TOUS les logs d'une même requête, automatiquement. C'est magique pour le debugging.

// LoggingFilter.java — Injecter un traceId dans chaque requête HTTP @Component public class LoggingFilter implements Filter { @Override public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) { String traceId = UUID.randomUUID().toString(); MDC.put("traceId", traceId); // Attaché à CE thread MDC.put("userId", getCurrentUserId()); // Tous les logs auront ces champs try { chain.doFilter(req, res); // Traiter la requête } finally { MDC.clear(); // TOUJOURS nettoyer après (thread reuse) } } } // Maintenant tous les logs de cette requête auront automatiquement traceId + userId !

// 04 — Monitoring Deep Dive

Le Monitoring avec Micrometer + Prometheus

Comment exposer, collecter et visualiser les métriques de ton app Spring Boot.

01

🎯 Spring Boot Actuator Auto-configuration

Spring Boot Actuator expose des endpoints HTTP qui donnent l'état de santé de ton app. C'est le point de départ de tout monitoring.

// application.yml management: endpoints: web: exposure: include: health, metrics, prometheus, info endpoint: health: show-details: always metrics: export: prometheus: enabled: true
🔗
/actuator/health → État de santé (UP/DOWN, DB, Redis…)
/actuator/metrics → Liste de toutes les métriques
/actuator/prometheus → Format que Prometheus scrape
/actuator/info → Version, git commit…
# Exemple réponse /actuator/health { "status": "UP", "components": { "db": { "status": "UP" }, "redis": { "status": "UP" }, "diskSpace": { "status": "UP" } } }
02

📏 Micrometer — Créer tes propres métriques Custom Metrics

Micrometer est à Java ce que SLF4J est aux logs — une façade universelle pour les métriques. Tu instrumentes ton code une fois, et ça marche avec Prometheus, Datadog, CloudWatch, etc.

// OrderService.java — Métriques métier @Service public class OrderService { private final Counter ordersCreated; private final Counter ordersFailed; private final Timer orderProcessingTimer; private final Gauge pendingOrders; public OrderService(MeterRegistry registry) { // Counter — compte des événements this.ordersCreated = Counter.builder("orders.created.total") .description("Total des commandes créées") .tag("currency", "MAD") .register(registry); // Timer — mesure la durée d'une opération this.orderProcessingTimer = Timer.builder("orders.processing.duration") .description("Temps de traitement d'une commande") .register(registry); // Gauge — valeur actuelle (comme un thermomètre) this.pendingOrders = Gauge.builder("orders.pending", orderRepository, repo -> repo.countPending()) .register(registry); } public Order processOrder(OrderDto dto) { return orderProcessingTimer.recordCallable(() -> { try { Order order = createOrder(dto); ordersCreated.increment(); // +1 à chaque succès return order; } catch (Exception e) { ordersFailed.increment(); throw e; } }); } }
03

🔥 Prometheus + Grafana Time-Series DB

Comment Prometheus fonctionne

Prometheus scrape (tire) les métriques de ton endpoint /actuator/prometheus toutes les N secondes. Il stocke tout en time-series (valeur + timestamp + labels).

# prometheus.yml scrape_configs: - job_name: 'spring-boot-app' scrape_interval: 15s # scrape toutes les 15s static_configs: - targets: ['localhost:8080'] metrics_path: '/actuator/prometheus'

Requêtes PromQL (Prometheus Query Language)

# Taux d'erreurs HTTP sur 5 minutes rate(http_server_requests_seconds_count{status="500"}[5m]) # Percentile 95 de latence histogram_quantile(0.95, http_server_requests_seconds_bucket) # Commandes créées par minute rate(orders_created_total[1m]) * 60 # Mémoire JVM utilisée en MB jvm_memory_used_bytes{area="heap"} / 1024 / 1024

// 05 — ELK Stack

La Stack ELK — Elasticsearch, Logstash, Kibana

Le trio le plus puissant pour centraliser, indexer et visualiser des millions de logs.

🔍

E — Elasticsearch

Moteur de recherche distribué basé sur Lucene. Stocke les logs en JSON, les indexe et permet des recherches full-text en millisecondes. Tu peux chercher parmi des milliards de logs instantanément. C'est la base de données des logs.

Distributed Full-text search JSON storage
⚙️

L — Logstash

Pipeline de traitement des données. Il reçoit des logs de multiples sources (fichiers, Kafka, Beats), les transforme (parser, enrichir, filtrer) et les envoie vers Elasticsearch. Pense-y comme un ETL pour les logs.

ETL Pipeline Grok Parser Multi-source
📊

K — Kibana

Interface visuelle pour Elasticsearch. Tu crées des dashboards, des visualisations, tu cherches et filtres dans tes logs en temps réel. C'est la fenêtre sur tes logs. Inclut aussi l'alerting et l'analyse ML.

Dashboards Real-time KQL Search

📤 Spring Boot → Logstash → Elasticsearch

Deux approches pour envoyer des logs depuis Spring Boot :

Option A — Logback Logstash Encoder (direct)

<!-- pom.xml --> <dependency> <groupId>net.logstash.logback</groupId> <artifactId>logstash-logback-encoder</artifactId> </dependency> <!-- logback-spring.xml --> <appender name="LOGSTASH" class="LogstashTcpSocketAppender"> <destination>logstash:5000</destination> <encoder class="LogstashEncoder"/> </appender>

Option B — Via Filebeat (recommandée en prod)

# Spring Boot écrit dans fichier JSON # Filebeat surveille ce fichier # Filebeat envoie à Logstash/Elasticsearch # filebeat.yml filebeat.inputs: - type: log paths: - '/logs/app*.log' json.keys_under_root: true output.logstash: hosts: ["logstash:5044"]

// 06 — Distributed Tracing

Tracing Distribué — Suivre une requête entre microservices

Quand tu as 10 microservices, comment savoir lequel est lent ? C'est le rôle du distributed tracing.

🕸️ Spring Cloud Sleuth + Zipkin / Jaeger

Chaque requête reçoit un TraceId unique. Chaque opération au sein de cette requête est un Span. En collectant tous les spans, tu peux visualiser le chemin complet d'une requête sur un graphique flame chart.

// Avec Spring Boot 3 + Micrometer Tracing (remplace Sleuth) // pom.xml <dependency> <groupId>io.micrometer</groupId> <artifactId>micrometer-tracing-bridge-otel</artifactId> </dependency> <dependency> <groupId>io.opentelemetry</groupId> <artifactId>opentelemetry-exporter-zipkin</artifactId> </dependency> // application.yml management: tracing: sampling: probability: 1.0 # 100% des requêtes tracées (réduire en prod) spring: zipkin: base-url: http://zipkin:9411 // Chaque requête aura maintenant dans ses logs: // traceId=abc123, spanId=def456 — propagés automatiquement !
🔗
W3C TraceContext : Les headers HTTP traceparent et tracestate propagent le traceId entre services automatiquement. Si ton Service A appelle Service B via HTTP, les deux partagent le même traceId — tu vois la chaîne complète.

// 10 — Roadmap de Construction

Comment construire ce projet — étape par étape

  1. Setup Spring Boot + Actuator + Logback JSON

    Créer l'app Spring Boot de base. Ajouter Actuator, configurer Logback pour sortir du JSON structuré avec SLF4J. Ajouter MDC Filter pour injecter un requestId dans chaque requête. Vérifier les logs en console d'abord.

  2. Ajouter Micrometer + métriques custom

    Ajouter la dépendance Micrometer Prometheus. Créer des Counters, Timers et Gauges sur tes opérations métier. Vérifier sur /actuator/prometheus que tes métriques apparaissent au bon format.

  3. Docker Compose — Lancer Prometheus + Grafana

    Écrire le docker-compose.yml avec Prometheus et Grafana. Configurer Prometheus pour scraper ton app. Importer un dashboard Grafana existant (Spring Boot Statistics). Voir tes premières métriques sur un dashboard.

  4. Stack ELK — Elasticsearch + Logstash + Kibana

    Ajouter ELK au docker-compose. Configurer Logback pour envoyer les logs JSON vers Logstash via TCP, ou utiliser Filebeat. Créer un index pattern dans Kibana. Faire ta première recherche dans les logs.

  5. Distributed Tracing — Zipkin / Jaeger

    Ajouter Micrometer Tracing + OpenTelemetry exporter. Lancer Zipkin en Docker. Créer plusieurs services qui s'appellent entre eux via RestTemplate ou WebClient. Visualiser une trace complète multi-services dans Zipkin UI.

  6. Alerting — Grafana AlertManager

    Configurer des règles d'alerte dans Grafana (taux d'erreurs > 5%, latence p95 > 500ms). Configurer les notifications (Slack webhook, email). Simuler une erreur pour tester la réception d'alerte.

  7. Reporting — Rapports automatiques

    Créer un service Spring Boot avec @Scheduled qui génère des rapports PDF/Excel à partir des données Elasticsearch. Envoyer par email. Ou utiliser Grafana Reporting (version Enterprise) pour exporter des dashboards en PDF.
